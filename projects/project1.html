Securing an Application

Summary

Designed the user authentication UI for our mobile application so that it could be added seamlessly into a customer's existing environment. Based on this design, refactored our web application user authentication to be consistent with the mobile app.

Roles

customer research, design, mock-up, css development

For this project, we needed to build user authentication into our mobile application so that it could co-exist in our customer's existing environment. To achieve this, we met virtually with the customer to understand their business, technical, and user requirements. Since, this customer already had branded mobile applications, we needed to design a solution that met both our company's business goals as well as the customer's business and technical requirements.

This project had two goals. The first goal was to enhance the security of an existing mobile application to meet the customer's security requirements. The second goal was to carry over these security enhancements over to an existing desktop application so that both applications shared the same security model. The security enhancements included:

  • the forgotten password scenario
  • setting up security challenge questions
  • answering challenge questions
  • allowing for and enforcing a customizable set of password rules

Although the mobile solution was targeted at a specific customer, the design needed to flexible and configurable enough to work in the general product which was used by a variety of different customers.

Mobile Application

first phase of the project was to enhance the mobile application with the security enhancements. To achieve this, we researched at a high level how the customer currently handled security, what their back-end systems required, and how our mobile application would interact with it. We reviewed the UI for their other mobile applications to understand the user journery scenarios to complete application tasks. In one case, we suggested a design improvement to the UI for changing passwords, which was accepted. The following assets were developed as part of the design:

  • flow diagram
  • UI wireframes

I developed the flow diagram by working through various scenarios required by the customer as well as additional ones needed by a general audience. Some of these scenarios included:

  • the forgotten password flow
  • setting up challenge questions with grace period
  • setting up challenge questions with no grace period
  • new user sign-in
  • resetting an existing user's password
By working through these and other scenarios, I began mapping out the screens and user and system interactions. Working with the Development team, I drafted, reviewed, and updated the flow until we had a model of the application. The result was the following flow diagram:

Around the same time as I was working on the flow diagram I also began constructing wireframes of the new and updated screens. As with any project, some of changes resulted in competing designs. To resolve design conflicts, we met as a team to understand the pros and cons of each one before deciding which approach to take. In some cases, we held design reviews with the customer to better understand their needs and requirements.

An example of a feature with competing designs was in the presentation of the challenge questions to the user. One approach was to display the configured number of security questions in dropdowns on a single screen, which another design was to adapt a wizard approach where each the user selected a question on one screen and then provided the answer on a separate screen.

In the end, we settled on the latter approach because that presentation was more consistent with other parts of the application and would provide more flexibility as to the number of question/answers needed and at what part of the system they were submitted to the authentication system. Once we had a set of wireframes, I assembled them in a PowerPoint to present to the customer. This presentation helped them understand not only how the new and update screens looked and behaved but also to ascertain how they would perform with their back-end authentication system. While the customer required some changes, they were pleased with result.
